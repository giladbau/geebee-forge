# Digest summary + citations improvement plan

## Goal
Improve the digest hero/notable UX so that:
1. hero summaries are concise but still informative enough to be useful
2. hero summaries include clickable inline links to the supporting sources shown below
3. source/citation labels are descriptive instead of generic junk like `github.com: GitHub` or bare `Project`

## Current context
- The current summary text is generated in `scripts/digest/shared/render.mjs`, mainly via `buildHeroSummary()`.
- The current UI renders summaries as plain text in `src/components/svelte/digest/DigestBrowser.svelte`, so links in summary text are not clickable.
- Source label cleanup is handled in `normalizeSource()` in `scripts/digest/shared/render.mjs`, but it still allows generic labels to leak through.
- The screenshot confirms the current bad states:
  - hero summary reads like a stiff generated blob
  - citations below are clickable, but the summary itself is not
  - at least one source label becomes essentially `github.com: GitHub`

## Proposed approach
### 1. Make summaries tighter and more editorial
- Replace the current boilerplate-heavy hero summary pattern.
- Keep enough detail to convey real information from 2–3 lead items.
- Remove weak metadata phrasing like `The cluster includes N more in-scope items across M source domains.` from the user-facing summary.
- Prefer a short report style over a shallow one-liner.

### 2. Add inline clickable links inside summaries
- Generate summary text with inline source references for the lead supporting items.
- Render those links safely in the Svelte UI instead of plain-text escaping them.
- Keep the existing source chips below as the structured evidence list.

### 3. Upgrade source label normalization
- Detect generic labels such as `GitHub`, `Project`, `Source`, `Link`, or raw host-prefixed placeholders.
- Fall back to a better item/source title when the source title is generic.
- Preserve short, human-readable labels that tell the reader what the click actually is.

## Likely files to change
- `scripts/digest/shared/render.mjs`
- `src/components/svelte/digest/DigestBrowser.svelte`
- `src/lib/digest/render.test.ts`
- possibly `src/lib/digest/*.test.ts` if UI rendering helpers need coverage

## Implementation steps
1. Add/adjust summary-building helpers in `render.mjs` so hero summaries become concise, informative, and link-aware.
2. Improve source normalization logic in `render.mjs` to demote generic labels and prefer meaningful titles.
3. Update `DigestBrowser.svelte` to render summary HTML safely for digest-generated inline links.
4. Add/adjust regression tests for:
   - concise informative summary shape
   - clickable inline links in summary content
   - better source labels for GitHub/project links
5. Run digest tests and relevant repo checks.
6. Review the diff, then push branch and open a PR.

## Validation
- `npm run digest:test`
- targeted test file(s) if useful
- `npm run check` if the UI rendering change touches Svelte/TS assumptions

## Risks / tradeoffs
- Rendering HTML in summaries can become unsafe if done lazily. Keep the supported output narrow and generated by our own renderer.
- Over-aggressive source-title cleanup can accidentally erase useful host context. Prefer readable titles, not blind prettification.
- Summary shortening should not collapse into vague fluff; the point is less padding, not less information.
